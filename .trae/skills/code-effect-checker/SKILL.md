# Code Effect Checker

用于排查“代码改了但效果不对/没生效”的问题，覆盖前端页面、接口联调、构建缓存、运行环境与日志定位。

## 何时调用

在以下场景优先调用本技能：

- 用户反馈“代码没生效”“页面效果不对”“改了接口但前端没变化”
- 出现“本地正常、运行后异常”或“偶发不一致”问题
- 需要快速判断是代码逻辑问题、环境问题、缓存问题还是数据问题

## 核心目标

1. 先复现：明确“期望 vs 实际”。
2. 再定位：确定问题层（前端/后端/接口/配置/缓存）。
3. 后修复：给出最小改动并验证。
4. 最后固化：补充检查步骤，避免回归。

## 标准排查流程

### 1) 复现与边界确认

- 记录问题页面/接口/功能入口。
- 明确期望行为与实际行为（最好可截图或日志片段）。
- 确认是否所有环境都复现（开发、本地、测试）。

### 2) 运行态检查

- 确认服务是否启动、端口是否正确、健康检查是否通过。
- 检查是否命中错误环境变量、错误 API 地址或代理配置。
- 排查热更新失效、构建产物未更新、浏览器缓存/Service Worker 干扰。

### 3) 代码路径校验

- 沿“入口 → 状态 → 请求 → 渲染”逐段确认逻辑是否执行。
- 检查条件分支、权限分支、空数据分支是否吞掉渲染。
- 对关键节点打印可审计日志（不要输出敏感信息）。

### 4) 数据与接口核对

- 核对请求参数、方法、响应结构、错误码处理。
- 检查字段重命名、类型变化（string/number/null）导致的兼容问题。
- 验证前后端约定是否一致（字段名、分页、时区、排序）。

### 5) 最小修复与验证

- 优先最小改动修复，不做无关重构。
- 修复后执行 lint / 测试 / 类型检查。
- 前端问题尽量通过可视化预览或页面健康接口再次确认。

## 常用诊断清单

- 是否改了文件但实际运行的是另一个目录/分支？
- 是否存在同名组件或路由覆盖？
- 是否因为缓存（浏览器/构建工具）导致旧代码仍在运行？
- 是否请求方法错误（GET/POST）导致接口未命中？
- 是否是 mock 数据掩盖了真实问题？
- 是否被 feature flag 或权限判断拦截？

## 输出模板（建议）

排查结论应包含：

1. **复现条件**：在哪个页面/接口、输入什么会触发。
2. **根因定位**：具体到文件与关键逻辑。
3. **修复方案**：最小改动说明。
4. **验证结果**：lint/测试/页面验证通过情况。
5. **防回归建议**：可自动化的检查项。

## 成功标准

- 问题可稳定复现并被修复。
- 修复后行为与“期望效果”一致。
- 质量门禁通过（lint、测试、类型检查）。
- 结论可复用，后续同类问题可按流程快速定位。
